## Compact Rules

### frontend-design
- Choose a clear visual direction and execute it intentionally; avoid generic AI/SaaS aesthetics.
- For EstuArriendo UI, honor the project design system: Brand Dark `#0F1F3D`, Brand Blue `#1B3A6B`, Brand Lime `#C8F135`, Plus Jakarta/Lato, light mode default with class-based dark mode.
- Prioritize production-quality React/Tailwind implementation with accessibility, responsive behavior from `xs` upward, and polished interaction states.

### vercel-react-best-practices
- For React/Next.js code, avoid request waterfalls: start independent async work early, use `Promise.all`, and place Suspense boundaries strategically.
- Reduce bundle cost: avoid barrel imports for large libraries, use dynamic imports for heavy components, and defer non-critical third-party code.
- Avoid unnecessary re-renders: derive state during render, use primitive effect dependencies, functional setState, stable defaults, and refs for transient values.
- Keep server/client boundaries lean: pass only fields needed by client components and authenticate Server Actions like API routes.

### systematic-debugging
- No fixes before root cause investigation. Reproduce, read errors fully, inspect recent changes, and gather evidence at component boundaries.
- Prefer diagnostic instrumentation over guessing when behavior crosses API/service/database or frontend/backend boundaries.

### cloudinary
- Use Cloudinary for image/video hosting, optimization, transformations, and CDN delivery.
- Signed uploads require `CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, and `CLOUDINARY_API_SECRET`; never commit credentials.

### backend-architecture
- Backend changes should preserve `routes → controllers → services → repositories → models` separation.
- Controllers handle HTTP concerns only; services own business logic; repositories own data access; models remain Sequelize schema definitions.

### cognitive-doc-design
- Lead with the answer, chunk dense content, prefer tables/checklists, and design docs for reviewer recognition over recall.

### branch-pr
- PRs must link an approved issue, use exactly one `type:*` label, and follow branch pattern `feat|fix|chore|docs|style|refactor|perf|test|build|ci|revert/<description>`.

### chained-pr
- Split PRs over the 400 changed-line review budget unless a maintainer approves `size:exception`; each slice must be autonomous and reviewable.

### work-unit-commits
- Commit by deliverable work unit, not file type. Keep tests/docs with the code they verify and maintain rollback clarity.

### comment-writer
- Human-facing comments should be warm, direct, short, actionable, and written in the thread language.
